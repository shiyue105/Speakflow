# 贡献指南

## PR 提交规范

### 粒度控制

1. **单一职责**：每个 PR 仅实现或修改单一功能模块
2. **细粒度提交**：尽可能提交小粒度 PR，便于审查与回溯
3. **大型功能拆分**：将复杂功能拆分为多个独立 PR 分步提交

### PR 内容要求

每个 PR 描述必须包含以下五个部分：

| 部分 | 说明 |
|------|------|
| 标题 | 一句话准确说明变更内容 |
| 功能描述 | 功能作用、场景、使用方式 |
| 实现思路 | 技术选型依据、核心算法/逻辑 |
| 测试方式 | 验证步骤、预期结果 |
| 备注 | 复用旧代码或参考第三方资料需注明来源 |

## 代码质量要求

- PR 合并后，主分支代码必须保持可运行状态
- 所有 commit 时间戳须在比赛周期内
- 禁止临近截止日期的"突击提交"
- 每个 PR 的 commit 需与自己 PR 一致，不得包含无关变更

## 分支策略

```text
main        ← 稳定分支，始终可运行
  ├─ feat/feature-name   ← 功能开发分支
  ├─ fix/bug-name        ← 修复分支
  └─ docs/topic          ← 文档分支
```

## 开发流程

```text
1. 从 main 创建功能分支
2. 在功能分支上开发并提交 commit
3. 发起 PR 到 main，填写完整描述
4. 自查通过后合并
5. 删除功能分支
```

## Commit 规范

建议使用 Conventional Commits 格式：

```
<type>: <description>

feat:     新功能
fix:      修复
docs:     文档更新
style:    代码格式（不影响逻辑）
refactor: 重构
test:     测试
chore:    构建/工具
```
